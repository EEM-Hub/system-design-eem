Data storage paradigm This article is about a storage system architecture. For the mode of data representation, see [Key–value pair](./Key–value_pair). [![](//upload.wikimedia.org/wikipedia/commons/thumb/7/74/BabbageKeyValueCard.tiff/lossless-page1-330px-BabbageKeyValueCard.tiff.png)](./File:BabbageKeyValueCard.tiff)A tabular data card proposed for Babbage's [Analytical Engine](./Analytical_Engine) showing a key–value pair, in this instance a number and its base-ten logarithm 

A **key-value database**, or **key-value store**, is a data storage paradigm designed for storing, retrieving, and managing [associative arrays](./Associative_array), a [data structure](./Data_structure) more commonly known today as a *dictionary*. Dictionaries contain a [collection](./Collection_(abstract_data_type)) of *[objects](./Object_(computer_science))*, or *[records](./Record_(computer_science))*, which in turn have many different *[fields](./Field_(computer_science))* within them. These records are stored and retrieved using a *key* that uniquely identifies the record, and is used to find the data within the [database](./Database).[[1]](./Key–value_database#cite_note-corbellini-1)[[2]](./Key–value_database#cite_note-autogenerated1-2)

 [![](//upload.wikimedia.org/wikipedia/commons/5/5b/KeyValue.PNG)](./File:KeyValue.PNG)A table showing different formatted data values associated with different keys 

Key-value databases differ from the better known [relational databases](./Relational_database) (RDB). RDBs pre-define the data structure in the database as a series of tables containing fields with well-defined [data types](./Data_type). Exposing the data types to the database program allows it to apply various optimizations. In contrast, key-value systems treat the value as opaque to the database itself, and typically support only simple operations such as storing, retrieving, updating, and deleting a value by its key. This offers considerable flexibility and makes such systems well suited to low-latency, high-throughput workloads dominated by direct key lookups, but less suitable for applications that require complex queries or explicit relationships among records.[[3]](./Key–value_database#cite_note-Gessert_et_al._2017-3)[[4]](./Key–value_database#cite_note-Grolinger_et_al._2013-4)

 

A lack of standardization, limited transaction support, and relatively simple query interfaces long restricted many key-value systems to specialized uses, but the rapid move to [cloud computing](./Cloud_computing) after 2010 helped drive renewed interest in them as part of the broader [NoSQL](./NoSQL) movement. Some [graph databases](./Graph_database), such as [ArangoDB](./ArangoDB),[[5]](./Key–value_database#cite_note-5) are also key–value databases internally, adding the concept of relationships (*[pointers](./Pointer_(computer_programming))*) between records as a first-class data type.[[4]](./Key–value_database#cite_note-Grolinger_et_al._2013-4)[[3]](./Key–value_database#cite_note-Gessert_et_al._2017-3)

 

## Types and examples

 

Key–value systems span a wide [consistency spectrum](./Consistency_model), from [eventually consistent](./Eventual_consistency) designs to strongly consistent or [serializable](./Serializability) ones, and some allow the consistency level to be configured as part of the trade-off against latency and availability.[[3]](./Key–value_database#cite_note-Gessert_et_al._2017-3)[[4]](./Key–value_database#cite_note-Grolinger_et_al._2013-4) Renewed interest in key–value and other NoSQL systems was driven in part by the demands of [big data](./Big_data), [distributed](./Distributed_data_store), and cloud applications. Their scalability and availability made them attractive for cloud data management, although limited transaction support, low-level query interfaces, and the lack of standardization remained obstacles to wider adoption.[[4]](./Key–value_database#cite_note-Grolinger_et_al._2013-4) Some maintain data [in memory (RAM)](./In-memory_database), while others employ [solid-state drives](./Solid-state_drive) or [rotating disks](./Hard_disk_drive).[[1]](./Key–value_database#cite_note-corbellini-1)

 

Some key–value systems add additional structure to their keys. For example, Oracle NoSQL Database organizes records using composite keys with "major" and "minor" components, an arrangement that Oracle compares to a directory-path structure in a file system. More generally, however, key–value stores are defined by their use of unique keys associated with opaque values and by their emphasis on simple key-based operations.[[6]](./Key–value_database#cite_note-6)

 

Unix included *[dbm](./DBM_(computing))* (database manager), a minimal database library written by [Ken Thompson](./Ken_Thompson) for managing associative arrays with a single key and [hash](./Hash_table)-based access. Later implementations and related libraries included sdbm, GNU dbm (gdbm), and [Berkeley DB](./Berkeley_DB).[[7]](./Key–value_database#cite_note-Seltzer_and_Yigit_1991-7)

 

A more recent example is [RocksDB](./RocksDB), a persistent key–value storage engine developed at Facebook and designed for large-scale applications.[[8]](./Key–value_database#cite_note-Dong_et_al._2021-8) Other examples include in-memory systems such as [Memcached](./Memcached) and [Redis](./Redis), and persistent systems such as Berkeley DB, [Riak](./Riak), and [Voldemort](./Project_Voldemort).[[4]](./Key–value_database#cite_note-Grolinger_et_al._2013-4)
